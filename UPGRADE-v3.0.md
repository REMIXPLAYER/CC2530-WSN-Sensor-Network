# CC2530 WSN v3.0 —— 升级计划与实现状态

> 标记说明：✅ 已实现 | 🔵 部分实现/改动 | ❌ 未实现/放弃 | ➕ 新增（计划外）

---

## 一、目标

去掉所有硬编码地址。传感器广播自动发现接收节点，P2P 直传数据。
PC 网关通过串口连接接收节点，负责云端上传和指令下发。

---

## 二、系统架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        云端服务器                                │
│                   (存储 + 用户指令UI)                            │
└──────────────┬──────────────────────────┬───────────────────────┘
               │ JSON/HTTP/MQTT           │ WebSocket/轮询
               ▼                          ▼
┌──────────────────────────┐  ┌───────────────────────────────────┐
│      PC 网关程序          │  │      CC2530 接收/透传节点        │
│  解析串口数据 → 云        │◄─┼── CH24: P2P 收传感器数据 → 串口 │
│  云指令 → 串口发送        │──►── CH24: 串口指令 → 广播 0xFFFF   │
│  维护节点列表             │  │  CH24: 周期 {TYPE=RECV} 信标     │
└──────────────────────────┘  └───────────────────────────────────┘
         ▲ 串口 UART              ▲ CH24 P2P直传  ▲ CH24 广播
         │                 ┌──────┴──────┐  ┌──────┴──────┐
         │                 │ DHT11       │  │ 光敏        │
         │                 │             │  │             │
         │                 │ RX窗收指令  │  │ RX窗收指令  │
         │                 │ 执行后P2P ACK│  │ 执行后P2P ACK│
         │                 └─────────────┘  └─────────────┘
```

> 🔵 架构调整：**单信道 CH24** 替代计划的双信道方案。数据和控制在同信道时分复用，CSMA-CA 处理冲突。

---

## 三、信道复用优化

| 状态 | 说明 |
|:----:|------|
| ❌ | **放弃双信道方案**。改为单信道 CH24 时分复用，理由：数据占空比低（~1%），CSMA-CA 退避足够覆盖偶发冲突。双信道增加节点复杂度（注册时需频点扫描），且不解决控制指令在传感器 RX 窗外的丢包。指令可靠性通过传感器 P2P ACK 机制保证（见 2-5 新增）。 |

---

## 四、通信设计决策

### 传感器数据：P2P 直传 CH24（可靠）

| 状态 | 项目 | 说明 |
|:----:|------|------|
| ✅ | 信道 | CH24 |
| ✅ | 方向 | 传感器 → 接收节点 |
| ✅ | 方式 | P2P，ACK 确认，5 次重试 CSMA-CA |
| ✅ | 格式 | `{Humidity=%u, Temp=%u, D=%d, MAC=...}` |

### 控制指令：广播 CH24 + MAC 自比对（零配置）

| 状态 | 项目 | 说明 |
|:----:|------|------|
| ✅ | 信道 | CH24（单信道，非计划的 CH25） |
| ✅ | 方向 | PC → 串口 → 接收节点 → 广播 0xFFFF → 传感器自比对 |
| ✅ | 方式 | 广播，无 ACK，不重试 |
| ✅ | 格式 | `{LED=1, MAC=00:12:4B:...}` |
| 🔵 | 命名 | 计划 `SEND=` 改为 `STATUS=`（更语义化） |
| ✅ | 接收节点 | 不解析、不查表、不记忆任何节点信息，纯透传 |

### 地址方案

| 状态 | 角色 | 短地址 | 来源 |
|:----:|------|--------|------|
| ✅ | 传感器 | `0x0001` | 统一占位（计划为 MAC 派生，实际统一 0x0001 更安全无冲突） |
| ✅ | 接收节点 | `0x4102` | 传感器通过信标 `basicRfReceiveAddress()` 获取 |
| ✅ | 信标/控制 | `0xFFFF` | IEEE 802.15.4 广播地址 |

> 🔵 传感器短地址从计划的 `(macAddr[1]<<8) | macAddr[0]` 改为统一 `0x0001`，避免与接收节点(0x4102)/广播(0xFFFF)的万分之一点六碰撞风险。节点识别完全依赖 payload 中 MAC 字符串。

## 五、数据包与注册

### 注册包 `{REG=...}`

| 状态 | 项目 | 说明 |
|:----:|------|------|
| ✅ | 格式 | `{REG=MAC:00:12:4B:...}` |
| ✅ | 方式 | P2P + ACK + CSMA-CA |
| ✅ | 结果处理 | 检查 `regResult`，成功打印 `{reg=OK, MAC=...}`，失败打印 `{reg=RETRY}` |

> 🔵 计划中无重试/失败处理，实际实现更健壮。

### 数据包

| 状态 | 格式 |
|:----:|------|
| ✅ | DHT11: `{Humidity=%u, Temp=%u, D=%d, MAC=%02X:...}` |
| ✅ | 光敏: `{Light=%d, D=%d, MAC=%02X:...}` |

### 控制包

| 状态 | 格式 |
|:----:|------|
| ✅ | `{LED=1, MAC=...}` / `{LED=0, MAC=...}` |
| 🔵 | `{STATUS=1, MAC=...}` / `{STATUS=0, MAC=...}`（计划名 `SEND=`） |

### ➕ 新增：指令 ACK 包（计划外）

| 状态 | 格式 | 说明 |
|:----:|------|------|
| ➕ | `{CMD=SUCCESS, LED=1, MAC=...}` | LED 亮确认 |
| ➕ | `{CMD=SUCCESS, LED=0, MAC=...}` | LED 灭确认 |
| ➕ | `{CMD=SUCCESS, STATUS=1, MAC=...}` | 恢复采集确认 |
| ➕ | `{CMD=SUCCESS, STATUS=0, MAC=...}` | 暂停采集确认 |

传感器执行指令后 P2P 回传 ACK 到接收节点，接收节点串口输出。解决了广播无 ACK 导致 PC 网关无法确认指令是否真正执行的问题。

## 六、端到端工作流

| 状态 | 步骤 | 节点 | 信道 | 动作 |
|:----:|------|------|------|------|
| ✅ | 1 | 接收 | CH24 | 周期广播 `{TYPE=RECV}` 信标（每 2 秒） |
| ✅ | 2 | 传感器 | 扫描 11–26 | 上电扫描，收到信标 → `basicRfReceiveAddress()` 获取接收地址 |
| ✅ | 3 | 传感器 | CH24 | 发 `{REG=MAC:...}` 注册包（P2P + CSMA-CA + ACK） |
| ✅ | 4 | 传感器 | CH24 | P2P 直传数据（含 D+MAC，ACK 可靠） |
| ✅ | 5 | 接收 | CH24 | 收到数据 → 串口输出 |
| — | 6 | PC 网关 | 串口 | 解析 → 维护节点表 → 上传云端（PC 网关独立开发） |
| — | 7 | 云端 | — | 用户选节点 + 操作 → 下发指令（云端独立开发） |
| — | 8 | PC 网关 | 串口 | 格式化 `{LED=1, MAC=...}` → 发送（PC 网关独立开发） |
| ✅ | 9 | 接收 | CH24 | ISR 收到 `}` → `cmdReady=1` → 主循环 `broadcastCmd()` → 广播 0xFFFF |
| ✅ | 10 | 传感器 | CH24(RX窗) | `strstr` 比对 MAC → 匹配则执行 |
| ➕ | 11 | 传感器 | CH24 | 执行后 P2P 回传 `{CMD=SUCCESS,...}` → 接收节点串口输出 |

## 八、技术问题与解决

| # | 问题 | 严重 | 方案 | 状态 |
|---|------|------|------|:----:|
| A | 广播 ackRequest=TRUE → 永远 FAILED | 🔴 | 广播前 `ackRequest = FALSE`，完后恢复 | ✅ |
| B | 串口指令 ~44 字符，FIFO 仅 1 字节 | 🟡 | ISR 环形缓冲 + `}` 分隔 | ✅ |
| C | 广播时串口继续收字节 | 🟡 | 环形缓冲解耦 ISR 和主循环 | ✅ |
| D | MAC `strstr` 可靠性 | 🟢 | 全球唯一，大小写一致 | ✅ |
| E | 单信道控制指令命中率低 | 🟡 | ❌ 双信道放弃 → ✅ 改为传感器 P2P ACK 确认 | 🔵 |
| ➕ | 传感器 RX 缓冲区太小 | 🔴 | `pRxData[16]` → `[64]` + 边界守卫 | ✅ |
| ➕ | 广播后无法确认执行成功 | 🔴 | 传感器 P2P 回传 `{CMD=SUCCESS,...}` | ✅ |
| ➕ | 光敏双重 ADC 读取 | 🟡 | 单次读取，值透传给 LCD 和 RF | ✅ |
